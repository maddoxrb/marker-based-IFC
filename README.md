# Speech-Gated IFC in AR

# Project Goal
The goal of this project is to implement an Information Flow Control system for Augmented Reality platforms in which physical markers enforce confidentiality polcies over associated virtual content. In a manner similar to the Decentralized Label Model, a formal security label will be attatched to each marker in an environment that dictates what virtual content can legally be displayed to the current user based on that user's permission level and user group. Likewise, the system will allow controlled movement of virtual objects between markers in a manner such that objects may only be moved to markers of equal or greater restrictiveness. With hands free applications in mind, the system will implement identity verification before sensitive data may be displayed at a marker via voice identification. To prevent potential issues concerning malicious pre-recorded voice clips, each marker will display a randomized pin code that must correctly read out loud.

Existing research on privacy and security in Augmented Reality systems largely focuses on methods to shield user privacy from invasive applications with access to potentially sentive data streams such as video and audio recording. Research on intra-user privacy in shared AR environments is far more limited, and generally omits any strong formal security framework to address hierarchical access control. Additionally, limited literature discusses access control for physically-bound virtual objects anchored via markers. I attempt to address this gap by implementing a system for formal and compositional Information Flow Control in marker-based AR environments.

# Technical Approach
I plan to implement this system as an iOS based AR application utilizing SwiftUI, RealityKit, and ARKit. The project will rely heavily on AR marker detection to recognize predefined markers placed in the environment. I'll utilize Apple's ARKit for marker detection and session management, and RealityKit for simple demo virtual objects. SwiftUI will be used to make the static frontend components, this will include the ability to uplaod custom markers, the ability to add confidentiality policies to markers, and status information (policy enforcement, last recognized marker, etc). Additionally, I will utilize SpeakerKit, an open source voice identification SDK for Swift, to perform voice recognition for identity verification. To verify if a code is correctly read, I will utilize Apple's SFSpeechRecognizer to transcribe icnoming audio in real time and confirm matching sequences. The full workflow is described in the model below, in short the application:

1. Runs a constant AR view that streams camera and audio input
2. When a marker is detected, the system loads the attatched policies and generates and displays a pin code
3. Under a ten second time frame the system observes audio input and attempts to:
   (1) Use SpeechKit to identify the speaker and load their clearance level and
   (2) Monitor SFSpeechRecognizer transcript for the correct pin code
4. If either identification or transcription fails reset pin code
5. Otherwise, compare the clearance level of the identified user to the attatched marker policy, display sensitive marker-gated virtual object only if acceptable

# Novelty
Existing work on security in AR is still pretty limited, and most of it focuses on privacy boundaries between applications and users (i.e, what an app can see or access). When access control is discussed, it typically assumes a standard model where each user is already authenticated through an account tied to the device. That assumption breaks down in shared-device settings, which are common in AR (headsets, tablets, or demo environments), where multiple users interact with the same system without persistent identity.

There is also very little work on controlling access to virtual content anchored to physical objects. Most AR systems treat anchored content as universally visible once detected, rather than something that should be filtered or gated based on who is viewing it. This project focuses specifically on that gap: enforcing access control over AR-rendered data tied to real-world markers.

I combine three existing methodologies to create a novel solution: (1) physical context through marker detection, (2) identity through speaker verification, and (3) liveness through a spoken challenge-response (random code). This allows the system to verify not just who is interacting with the content, but also that they are present and responding in real time, without relying on traditional login flows.
The novelty stems from putting together AR tracking, voice biometrics, and challenge-response verification into a single pipeline that works in a shared-device environment. It also focuses on enforcing information 
flow policies in physical space, where access decisions depend on both the user and the object being viewed.

# Evaluation
This system only works if it can reliably make the correct access decision under repeated trials and realistic edge cases. For this project, success means the AR application consistently renders sensitive content only when all required conditions are met: the correct marker is detected, the speaker is correctly verified, and the spoken challenge code is correctly matched within the allowed time window. If any of these checks fail, the system should deny access and render only the restricted view.

To evaluate the success of this system, I will create a set of trials that varies three factors: authorized vs unauthorized speaker, correct vs incorrect spoken code, and clean vs nosiy capture conditions. For each trial, I will measure true accepts, true rejects, false accepts, and false rejects, with the most important metric being the false accept rate. The system would be successful if it has a very low false accept rate, ideally zero, while still maintaining a high true accept rate for legitimate users.

I will also evaluate the system's runtime behavior itself by checking whether the system correctly updates rendered output when the marker is briefly lost and reacquired, whether access decisions stay synchronized with the currently displayed marker, and how long it takes from marker detection to final rendering decision. Ideally the system shoukd remain stable during repeated interactions and not accidentally leave sensitive content visible after tracking loss or failed verification. 

# Milestones and Contingencies
For the MVD, my goal is to have a simplified version of the full pipeline. The MVD should be able to detect pre-loaded markers with attatched security policies, gate sensitive content, perform voice identification, and enforce access control based on the results. At this stage, I will not include the pin code generation, instead I will focus on the bulk difficulty of the system which will be the access control logic. The MVD should have a few pre-loaded user voices with differing permissions levels and the marker policies should be structured to test access for each user. The actual virtual objects themselves can be simple 3d shapes, the UI kept to Swift basics, mainly focusing on verification and policy enforcement logic correctness. 

I plan to build this in a few stages. First, I will get ARKit marker detection and stable object rendering working, since everything depends on reliable tracking. Then I'll integrate the policy layer so that different markers correspond to different access levels and rendering states. Then Ill add audio recording input and set up SpeakerKit for identity verification. With the MVD complete, I will then implement pin code generation and add the speech recognition flow to transcribe the challenge code.

The hardest part will likely be keeping all of these components synchronized in real time. Since the AR tracking, audio capture, transcription, and speaker verification will all operate asynchronously I could see it being easy for the system state to get out of sync (a marker disappearing while audio is still being processed, etc). Making sure that each access decision corresponds to the correct marker, the correct challenge, and the correct time window will be the main technical challenge. I will also need to put thought into handling edge cases and noisy inputs.

If I fall behind, I would likely reduce the number of markers and supported access levels to just one or two. I also plan to keep the visual side minimal and use generic assets. If need be, the pin code verification logic, while ideal for security strength, could be dropped and the proof-of-concept would remain valid. Ultimately th priority is to ensure that the system performs policy-aware rendering based on identity even if some of the more advanced features are reduced.

This MVD lays out the core functionality and testing suite for the project. In particular, this MVD adds the following:
1. Three predefined access control groups defined in increasing restrictiveness: public -> employee -> admin
2. Implements a basic UI and control flow (credit https://github.com/pinlunhuang/Voice-Recorder) to allow users to set up new user profiles under one of the three access control groups, and add 1+ 10 second voice clips for authentication.
3. Adds three initial built-in markers, MarkerA, MarkerB, MarkerC (see assets/ARResources) with predefined confidentiality policies as public, employee, admin respectively
4. Adds in a basic UI to upload a custom marker and attatch a confidentiality policy, and a policy management UI
5. Adds a homescreen with an active ArKit scene camera view and a basic swiftUI control panel below 
6. ArSession handling to detect registered markers, enforce confidentiaity policies, and display legally visible data

The current workflow operates as such:

AR session setup starts in MarkerTrackingView.swift, which implements the home UI combining both SwiftUI controls with the ARSession. This creates a custom ARKit scene coordinator, ARSceneCoordinator.swift, which is responsible for event handling, specifically marker detection. Through the method processImageAnchors(), the coordinator consistently checks if any of the stored markers are visible within the session, if a marker is detected, the coordinator then calls handleImageAnchor() which uses the marker name to update tracking state, attach the corresponding RealityKit anchor, and notify the application model that a marker scan has occurred.

Marker policy enforcement is then handled by AppModel.swift together with ARSceneCoordinator.swift. AppModel stores the attatched confidentiaity policy for each marker which includes the minimum access level required and the object assigned to that marker. The coordinator reads that policy when deciding how the marker’s content should be rendered. If the current access level is not high enough, the coordinator renders a placeholder red cube, otherwise the attatched object is rendered.

If the controls have been switched to voice authentication mode, then instead the coordinator will call AppModel.handleMarkerScan(), which starts the voice auth. That workflow is implemented across AppModel.swift and SpeakerAuthentication.swift. AppModel manages the state and orchestration, while SpeakerAuthentication.swift contains the recording and speaker-matching logic. A short voice sample is captured after the marker is scanned, and then using PicoVoice it is compared against stored speaker profiles. If a match is found, the matched speaker’s access level becomes the effective access level used by the rest of the system.

Although working in its current state, the system seems to experience some runtime lag spikes when more than one marker is introduced into frame, which I am currently working on pinpointing. I suspect somewhere the system is continually bouncing back and forth between marker scans leading to the lag spikes, but I have yet to confirm where this might be happening. As such, I believe I will have to remove object transfers (drag and drop) between markers from the project scope. While ideal for showcasing potentia use cases, the movement of objects ultimately does not effect the novelty of this project, which focuses more on voice based authentication at the marker-level. Moving forward, aside from testing and minor improvements, I need to implement the pin-code verification. This will require (1) attatching logic to the marker detection flow to randomly generate and display a pincode before authentication, and (2) use apple's speech recognition to transcribe the captured voice sample and determine if the correct pin was spoken. I am concerned that speaking the code alone may not be enough for the voice identification to work, so I will need to test this beforehand and potentially shift my concept accordingly. 


# Final Report

## Evaluation and Results
To evaluate the effictiveness and robustness of the system, I used a controlled experimental set up that was designed to isolate the effects of the speaker identification system, the spoken code matching, the authorization lattice, and the environmental recording conditions. The study included 24 trials that spanned all combinations of the following variables:
1. Speaker Authorization: Did the speaker have the required permission level to view the marker data (authorized vs. unauthorized)
2. Code Correctness: Was the correct pin code spoken at authentication-time (correct vs. incorrect)
3. Capture Condition: The quality of the audio recording environment (clean vs. noisy)
4. Speaker Identity: Three distinct speakers (Maddox, Speaker A, Speaker B)

Each trial entailed a realistic interaction where an end user attempted to access a marker by speaking a short phrase ("I am speaker X, the code is Y") at authentication time for a given marker. The trial used three speakers, firstly myself, and secondly two speakers of distinct tone generated via ElevenLabs(https://elevenlabs.io/app/speech-synthesis/text-to-speech). To test robustness to environmental capture quality, trials were performed both in clean silent conditions, as well as audio captures where a realistic busy background ambient noise was played at a consistent volume through a nearby speaker (https://www.youtube.com/watch?v=h2zkV-l_TbY). For each combination of variables, a trial was performed for each speaker identity whereupon the system would produce a binary decision of allow or deny based on the speaker verification and pin code matching. I collected overall results across these trials and compared them against the expected system decision, evaluating specifically for the following metrics:
1. Total Accuracy: Percentage of trials where the system performed exactly as expected
2. Speaker Identification Accuracy: Percentage of trials where the correct speaker was identified
3. Code Matching Acccuracy: Percentage of trials where spoken code was registered correctly
4. False Accept Rate (FAR): Percentage of trials where an unauthorized user was given access to confidential data
5. False Reject Rate (FRR): Percentage of trials where an anuthorized user was rejected from legal marker access

The results of the evaluation show that the system is generally performant with some limitations. Firstly, from a security perspective, the system is very conservative with a 0% FAR, this is ideal in a security scenario, we would rather false rejects rather than false accepts. With that being said however, we did find a false reject rate of 16.7% which is significant and likely intolerable in deployment. This case from 1 of 6 trials where the system was unable to identify the speaker in a noisy environment. Across all trials, we observed 100% accuracy in the code matching, all correctly spoken codes were regsitered as correct and all incorrectly spoken codes regsistered as incorrect. Apple's SFSpeechRecognizer is very robust and provides reliable speaker transcription across audio environments, it is certainly not the limiter of the system. Instead, examining the failure trials reveals that the real bottleneck of the system is the voice identification. In in all of the failed trials, the failure was a result of the system producing a confidence level for the given speaker below the tolerable threshold, and of those four failed trials 3 were observed in noisy conditions. This indicates two things, firstly speaker identification through PicoVoice Eagle is generally reliable when audio is captured in clean environments, but more importantly this performance degrades significantly across noisy and un-ideal environments. This is an important finding, as it limits potential deploybility in different settings, particularly in industrial settings that may be far more prone to loud environmental noise. Potential further work may involve improved identification methods or perhaps techniques to isolate speaker audio from ambient noise. In HMD environments, Shang et al. have shown extremely high accuracy (97%) voice identification methods for HMDs that utilize air and contact microphones for analysis, demonstrating feasible and promising directions to improve upon the bottleneck of this system.

It is important to note that this evaluation has several limitations. Firstly, due to timing constraints the evaluation was performed upon a minimal number of trials, future work would extend these studies to discern more concrete and definitive evaluation results. Secondly, of the three speakers used in the study two of those used were artificially generated. While artificially generated voices have become highly advanced, they still have certain characteristics that are clearly differentiable from genuine human speakers, which may limit the confidence in the observed results. Lastly, although best efforst were made to ensure constants were held across trials, there are some potential fluctations between experiments that may affect the results, such as speaker volume and proximity (particularly for the human speaker), and environmental audio disruptions.
